# 瀚铎智脑 (HydroMind)：面向水系统的Agent原生基础设施

> 产品概念白皮书 v0.1
> 2026-02-25 雷晓辉

---

## 一、核心命题

**未来的软件是给Agent用的。** 那么，水利行业的Agent需要一个什么样的"家"？

答案是：一台私有化部署的、始终在线的Agent基础设施——集成了水系统领域知识、多智能体协作引擎、和全链路研发工具链的"水利大脑"。

我们称之为 **瀚铎智脑（HydroMind）**。

---

## 二、产品定位

```
传统模式：人 → 操作软件 → 得到结果
HydroMind：人 → 定义意图 → Agent群协作 → 人验收结果
```

HydroMind 不是一个"给人用的App"，而是一个**给Agent群运行的操作系统**。人类通过自然语言（微信/钉钉/Web）下达任务，Agent群自主完成从文献调研、论文写作、水系统建模、调度优化到在环测试的全链路工作。

## 三、硬件底座：私有NAS

### 为什么必须私有化？

| 维度 | 云端Agent | 私有NAS Agent |
|------|----------|--------------|
| 数据主权 | 论文草稿、实验数据、调度策略全部上传第三方 | 所有数据不出局域网 |
| 持久运行 | 按token计费，长时间任务成本爆炸 | 24/7始终在线，边际成本趋零 |
| 上下文积累 | 每次会话从零开始 | Agent记忆持久化，知识持续积累 |
| 模型自由 | 受限于API供应商 | 本地模型+云端模型混合调度 |
| 安全合规 | 涉密项目数据出境风险 | 物理隔离，满足涉密要求 |

### 推荐硬件配置

```
基础版（课题组）：
  - Synology DS1621+ 或 QNAP TS-664
  - 64GB RAM + 2TB NVMe缓存
  - 本地推理：Mac Mini M4 Pro (48GB) × 2 作为推理节点
  - 网络：内网千兆，Tailscale穿透

进阶版（实验室/研究所）：
  - 专用服务器：双路EPYC + 256GB RAM
  - GPU推理节点：RTX 4090 × 2（本地中等模型）
  - NAS存储：100TB+ RAID阵列
  - 云端API作为overflow（Anthropic Claude / OpenAI）
```

**关键设计原则：本地优先，云端兜底。** 日常任务跑本地模型（Llama、Qwen），需要顶级推理能力时按需调用云端API。

## 五、六大Agent群详细设计

### 5.1 科研写作Agent群（HydroScribe）

这是已经验证过的成熟架构，从现有的多智能体写作系统演化而来。

```
Writer Agent × 9（按文体分工）
  ├── 专著章节Writer（长文、学术风格）
  ├── SCI论文Writer（英文、IMRaD结构）
  ├── 中文期刊Writer
  ├── 专利Writer（权利要求书风格）
  ├── 项目申报Writer
  ├── 技术报告Writer
  ├── 综述Writer
  ├── 科普Writer（公众号风格）
  └── 学位论文Writer

Reviewer Agent × 6（按维度分工）
  ├── 学术规范Reviewer（引用、格式、术语一致性）
  ├── 逻辑严密性Reviewer（论证链、因果关系）
  ├── 数据核查Reviewer（数字、图表、公式）
  ├── 语言质量Reviewer（中文/英文润色）
  ├── 查重风险Reviewer（表述原创性）
  └── 总编Reviewer（全局统稿、风格一致）

Utility Agent × 3
  ├── 文献检索Agent（Brave Search + 自引数据库）
  ├── 排版输出Agent（docx/LaTeX/pdf生成）
  └── 图表生成Agent（matplotlib/Visio/mermaid）
```

**核心Skill举例：**
```markdown
# SCI论文Writer Skill
- 严格遵循IMRaD结构
- 引用格式：优先自引lei-2025a至lei-2025d
- 数据主张必须有出处，否则标注 ⚠️待核实
- 每段不超过200词，每句不超过35词
- 禁止编造参考文献
```

### 5.2 建模仿真Agent群

```
建模Agent群
  ├── 物理建模Agent（PBM构建）
  │     Skill: Saint-Venant方程 · 水库水量平衡 · 管网水力
  ├── 简化建模Agent（SM构建）
  │     Skill: ID模型 · 传递函数辨识 · 降阶方法
  ├── 观测建模Agent（OSEM构建）
  │     Skill: 卡尔曼滤波 · 数据同化 · 状态估计
  ├── 参数率定Agent
  │     Skill: 遗传算法 · 贝叶斯优化 · 灵敏度分析
  └── 仿真执行Agent
        Skill: OpenModelica调用 · Simulink接口 · Python数值求解
```

**工作流示例：渠道ID模型辨识**
```yaml
pipeline: canal-id-model
steps:
  - agent: 物理建模Agent
    task: 根据渠道几何参数建立Saint-Venant方程
    output: physical_model.py
  - agent: 简化建模Agent
    task: 将分布参数模型简化为ID模型（积分延迟模型）
    input: physical_model.py
    output: id_model.py
  - agent: 参数率定Agent
    task: 用实测数据率定ID模型参数
    input: [id_model.py, field_data.csv]
    output: calibrated_params.json
  - agent: 仿真执行Agent
    task: 对比验证ID模型与实测数据
    output: validation_report.md
```

## 七、典型使用场景

### 场景1：一篇SCI论文的全自动化生产

```
雷老师（微信）: 写一篇关于大渡河梯级水电站群
    水光互补优化调度的SCI论文，投Water Resources Research

→ 项目管理Agent 分解任务
→ 文献情报Agent 检索最新相关文献（50篇）
→ 文献精读Agent 提取关键方法和数据
→ SCI论文Writer 生成初稿
→ 学术规范Reviewer 审查引用和格式
→ 逻辑严密性Reviewer 审查论证链
→ 数据核查Reviewer 校验所有数值
→ Writer 根据审稿意见修改（最多3轮）
→ 语言质量Reviewer 英文润色
→ 排版输出Agent 生成Word稿件
→ 雷老师收到微信通知：论文初稿已完成，请审阅

全程预计：4-8小时（传统方式：2-4周）
```

```
雷老师（钉钉）: 为胶东调水某段渠道设计MPC控制器，
    要求超调量<5%，调节时间<2小时

→ 物理建模Agent 读取渠道参数建立Saint-Venant方程
→ 简化建模Agent 辨识ID模型
→ 参数率定Agent 用历史数据校准

    附仿真结果、测试报告和设计文档

全程预计：1-2天（传统方式：1-3个月）
```

## 十、这个产品的本质

回到那篇公众号文章的核心论断：**未来的软件是给Agent用的。**

HydroMind 就是这个论断在水利领域的具体实践：

- **Skill不是文档，是Agent的App**——每个SKILL.md就是一个Agent可以"安装"的能力模块
- **Pipeline不是流程图，是Agent的工作流**——YAML定义的确定性流程，Agent自主执行
- **HydroOS不是操作系统界面，是Agent的教科书**——结构化的领域知识，Agent按需读取
- **NAS不是存储设备，是Agent的家**——24/7在线，记忆持久化，数据不出门

人类要做的只有三件事：**定义意图、设定边界、验收结果。**

其他所有事情——检索文献、撰写论文、建立模型、优化调度、测试验证、生成报告——都由Agent群在幕后完成。

这不是未来。这是现在就可以开始建设的东西。
