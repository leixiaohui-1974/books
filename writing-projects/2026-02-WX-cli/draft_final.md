# 大厂为什么都在做命令行 AI

2026 年 2 月，一个有意思的现象：三大 AI 实验室同时在推命令行工具。Anthropic 的 Claude Code，OpenAI 的 Codex CLI，Google 的 Gemini CLI。网页版聊天用得好好的，为什么要把 AI 塞进终端？

我最近亲身经历了一次才想通。

---

## 一个 1100 行文件的诞生

我在设计一个水网智能决策平台的技术架构。这不是写一段代码的事，而是一份包含五层架构、九类 MCP 工具、六种用户角色、四预闭环 Skill、分布式计算层和完整开发计划的系统设计文档。最终 1100 行，13 个章节。

整个过程用的是网页版 Claude，大约三轮迭代完成。第一轮出骨架，我说"加入 Skill 层和 Ray 分布式计算"，第二轮从三层变五层。我又说"按用户角色深化，加入水利部四预体系"，第三轮加了 300 行场景化设计。每一轮都是在上一轮的文件上精确修改，不是重写。

做完之后我才意识到：如果换成命令行工具，这个过程会更自然。

---

## 网页聊天的天花板

网页聊天的核心模式是"你说一句，我回一段"。写一篇文章、回答一个问题，这个模式足够了。但做系统设计时，问题变了。

你需要 AI 同时理解一个 800 行的文件、搜索最新的技术资讯、在特定位置做精确替换、验证修改后的结构是否完整。这些操作在网页版里全靠 AI 自己协调，用户看到的只是一轮一轮的对话。

命令行工具不同。它天然就生活在文件系统里。读文件、改文件、跑脚本、查 Git 历史，这些在终端里是原子操作。AI 不需要把文件"想象"成文本块粘在对话里，它直接操作源文件。

差别就像是：一个人对着电话描述一幅画让对方修改，另一个人直接坐在画布前动笔。

---

## 不是"编程工具"，是"工作环境"

很多人把 Claude Code 理解为"帮程序员写代码的工具"。这个理解太窄了。

命令行 AI 的本质是：AI 有了一个可以操作真实文件、运行真实命令的工作环境。程序员用它写代码，但科研人员同样可以用它管理论文工程、搭建仿真平台、设计系统架构。只要你的工作可以被分解成"在文件系统里做一系列操作"，命令行 AI 就比网页聊天强。

我设计水网平台的过程就是个例子。核心操作并不是写 Python 代码，而是迭代一份 Markdown 架构文档。但迭代过程需要精确定位修改位置、保持章节编号一致、同时搜索外部资料——这些能力在终端里是标配。

---

## 三家的思路其实不同

虽然三家都做了命令行工具，但思路差异很大。

Claude Code 押注"自主性"。它的 Agent Team 功能可以派出子智能体并行处理多个任务，像一个自带施工队的总工程师。优势是复杂任务的一次性完成率高。

Codex CLI 押注"安全"。开源、沙箱隔离、三级权限审批。每一步操作都需要你确认，像一个谨慎的合规官。适合对代码安全要求高的团队。

Gemini CLI 押注"性价比"。一百万 token 的上下文窗口加上每天一千次免费请求，直接把门槛降到零。适合先试水的团队，或者用来做规划和文档而非核心编码。

有意思的是，很多开发者不选一个，而是组合使用。用 Gemini 做规划和拆分任务，用 Claude Code 做核心实现，用 Codex 做调试和代码审查。这个分工模式和水利行业的分层架构异曲同工——不同层次用不同工具。

---

## 水利人应该怎么看

聊这些和水利有什么关系？关系大了。

水利部正在全面推进数字孪生流域建设，核心是"四预"——预报、预警、预演、预案。这四个环节的技术落地，本质上就是"在计算环境中执行一系列编排好的操作"：跑模型、比方案、出报告、生成预案。

命令行 AI 的思路，和水网自主运行平台的架构逻辑几乎一样。底层是固定的核心工具——仿真引擎、控制算法、预测模型。上层是灵活的智能编排——大模型理解用户意图，选择合适的工具组合，执行任务链。

Claude Code 通过 CLAUDE.md 文件来配置项目规则。水网平台通过 Skill 定义来封装固定工作流。Claude Code 的 Agent Team 并行执行多任务。水网平台用 Ray 并行推演多个调度方案。

同构的架构，不同的领域。

---

## 终端回来了

过去十年，软件界的趋势是"离开终端"。IDE 越来越重，浏览器编辑器试图取代本地开发。但 2026 年，终端重新成为引力中心。

原因很简单：AI 需要一个可以"动手"的地方。

网页聊天是"说"的界面。命令行是"做"的界面。当 AI 从回答问题进化到完成工作，它需要的不是更好的对话框，而是一个可以读写文件、运行程序、管理状态的工作空间。

终端恰好就是这样的地方。它已经存在了五十年，接口简单、功能完备、可编程、可组合。AI 不需要重新发明一个工作环境——只需要住进这个现成的。

---

下一次你看到技术团队讨论"要不要上 Claude Code"的时候，不妨换个角度想：这不是程序员的事，这是所有需要"AI 动手干活"的人的事。水利规划、工程设计、论文写作、数据分析——只要你的工作涉及文件和流程，命令行 AI 都可能是比网页聊天更好的选择。

毕竟，真正的工作不是靠聊天完成的。
