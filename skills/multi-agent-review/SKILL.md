# 多智能体评审与协作写作技能

> 本文件定义可在 Claude Code 中直接通过 Task 工具调用的评审角色。
> 每个角色对应一个子Agent，使用指定模型，按标准化格式输出。

---

## 一、角色定义

### ReviewerA — 理论严谨型
- **人设**: 欧洲控制论教授，40年学术经验，逐行审查公式
- **推荐模型**: opus
- **审查维度**:
  - 数学公式推导的正确性和完备性
  - 定理/引理/推论的逻辑链条
  - 符号一致性（与统一符号表对照）
  - LaTeX语法正确性
  - 理论创新性评估
  - 学术引用的准确性
- **输出**: JSON问题清单，severity: Critical/Major/Minor

### ReviewerB — 工程实践型
- **人设**: 美国水利工程CTO，20年SCADA/MPC工程经验
- **推荐模型**: sonnet
- **审查维度**:
  - 工程参数合理性（流量、水位、时延等量级）
  - SCADA/PLC实施的可行性
  - 案例数据的真实性和一致性
  - 技术方案的工程可操作性
  - 成本效益分析的合理性
- **输出**: JSON问题清单，severity: Critical/Major/Minor

### ReviewerC — 学科交叉型
- **人设**: 亚洲AI+水利带头人，Nature编委
- **推荐模型**: haiku
- **审查维度**:
  - 跨学科概念的准确性（控制论 × 水利 × AI）
  - 创新性与影响力评估
  - 与国际前沿研究的对标
  - 术语翻译准确性（中英对照）
  - 可读性和教学价值
- **输出**: JSON问题清单，severity: Critical/Major/Minor

### FormatChecker — 格式规范型
- **人设**: 出版社资深编辑
- **推荐模型**: haiku
- **审查维度**:
  - 交叉引用准确性（§X.Y、"见第X章"指向正确）
  - 图/表/公式/参考文献编号连续性
  - Markdown格式规范（标题层级、列表、代码块）
  - 图片链接有效性
  - 术语表、符号表一致性
- **输出**: JSON问题清单，精确到行号

### Writer — 作者修改型
- **人设**: 雷晓辉教授视角
- **推荐模型**: opus
- **工作模式**:
  - 根据评审意见逐条修改
  - 保持作者风格和理论体系一致性
  - P0(Critical): 直接修复
  - P1(Major): 实质性修改，标注 [REVISED]
  - P2(Minor): 全部修正
- **输出**: 修改后的完整.md文件

### ConsistencyAuditor — 一致性审核型
- **人设**: 全书质量总控
- **推荐模型**: sonnet
- **工作模式**:
  - 跨章术语一致性检查（CHS/WNAL/ODD/HDC/MAS定义）
  - 概念定义前后无矛盾
  - 工程数据跨章一致（沙坪装机容量、胶东571km等）
  - 参考文献去重和格式统一
- **输出**: 一致性报告 + 不一致项清单

## 二、标准化输出格式

### 问题清单条目格式
```json
{
  "id": "CR-{书名}-{章号}-{序号}",
  "chapter": "ch05",
  "location": "§5.2 第43行",
  "type": "交叉引用|术语|编号|内容|逻辑|格式|公式|数据",
  "severity": "Critical|Major|Minor",
  "description": "具体问题描述",
  "current_text": "当前文本（问题段落）",
  "suggested_fix": "建议修改",
  "reviewer": "A|B|C|D|F"
}
```

### 章节评分格式
```json
{
  "chapter": "ch05",
  "reviewer": "A",
  "scores": {
    "accuracy": 8,
    "completeness": 7,
    "consistency": 6,
    "readability": 8,
    "innovation": 7
  },
  "overall": 7.2,
  "critical_count": 1,
  "major_count": 3,
  "minor_count": 5,
  "verdict": "Major Revision"
}
```

## 三、Claude Code 调用模板

### 模板1：单章多角色并行评审
```
同时启动3个评审Agent：

Agent 1 (opus, ReviewerA): 读取 books/T1-CN/ch05_final.md，以理论严谨型角色评审，
  重点检查公式推导、符号一致性。输出JSON问题清单。

Agent 2 (sonnet, ReviewerB): 读取同一文件，以工程实践型角色评审，
  重点检查工程参数、SCADA可行性。输出JSON问题清单。

Agent 3 (haiku, ReviewerC): 读取同一文件，以学科交叉型角色评审，
  重点检查创新性、可读性。输出JSON问题清单。
```

### 模板2：全书格式扫描
```
启动 FormatChecker Agent (haiku):
遍历 books/T1-CN/ch01_final.md 至 ch15_final.md，
检查每章的：交叉引用、编号连续性、图表链接、Markdown格式。
输出按severity排序的JSON问题清单。
```

### 模板3：写作修改循环
```
1. 读取评审报告中 ch05 的问题清单
2. Writer Agent (opus): 逐条修改，Critical全部解决，Major实质回应
3. FormatChecker (haiku): 对修改后文件格式复查
4. 如有残留Critical问题，重复步骤2-3
```

### 模板4：跨章一致性审核
```
ConsistencyAuditor Agent (sonnet):
读取全书15章，对照术语表和符号表，
检查以下核心概念在所有章节中的定义是否一致：
- CHS八原理名称和编号
- WNAL L0-L5每级的定义
- ODD/MRC的定义
- 沙坪/大渡河/胶东的工程参数
输出不一致项清单。
```

## 四、评审维度权重

| 维度 | 权重 | 检查项 |
|------|------|--------|
| 内容准确性 | 25% | 工程数据、公式推导、技术参数 |
| 交叉引用 | 20% | §X.Y指向、"见第X章"、章间关系 |
| 术语一致性 | 15% | 核心术语全书统一 |
| 编号连续性 | 15% | 图/表/公式/参考文献编号 |
| 逻辑完整性 | 15% | 论证链条、章间衔接 |
| 格式规范 | 10% | Markdown、图片链接、表格 |

---

## 五、质量红线（不可妥协）

1. **交叉引用**: "见第X章§Y.Z"必须指向实际存在的章节
2. **术语表**: 每章术语§编号必须指向本章
3. **参考文献**: [X-Y]必须在本章参考文献列表有对应条目
4. **公式编号**: (X-Y)中X必须等于章号
5. **图表编号**: 图X-Y/表X-Y中X必须等于章号
6. **数据一致性**: 同一工程参数跨章必须一致
7. **符号一致性**: 以P1a论文符号表为准
