# 为什么所有 AI 公司都在做命令行工具

2025 年 4 月，OpenAI 发布了 Codex CLI。5 月，Anthropic 发布了 Claude Code。Google 紧随其后推出了 Gemini CLI。到 2026 年初，三大模型厂商全部拥有了自己的终端命令行工具。

这不是巧合。

---

我是在自己动手之后才真正理解这件事的。最近我在设计一个水利行业的多智能体平台，架构涉及仿真、辨识、控制、优化等十几个模块，还要对接国际上最新的 MCP 和 A2A 协议。这个系统用网页聊天根本做不了——它需要读文件、搜资料、写代码、反复修改，一轮接一轮。

我用 Claude Code 在终端里完成了整个过程。说一句"加入分布式计算层"，它就去读现有代码，搜 Ray 框架的最新文档，然后直接改提示词文件、调整目录结构、更新章节编号。全程没有复制粘贴，没有手动切换窗口。

这时候我才明白：命令行工具解决的不是"聊天"的问题，是"干活"的问题。

---

## 网页版的天花板在哪里

网页版聊天界面适合问答。你问一个问题，它给一个回答。即使加上文件上传和代码执行，本质上还是"一问一答"的循环。

但真正的工作不是一问一答。写一份完整的技术方案，需要先了解背景、再搜集素材、然后写初稿、跑检查脚本、修改、再检查。这个过程有十几个步骤，每个步骤的输出是下一个步骤的输入。

网页版做这件事，你得不停地复制粘贴——把上一轮的输出贴到下一轮的输入里。文件多了就乱，上下文断了就丢，窗口切多了就累。

命令行工具直接在你的电脑上运行。它能读你硬盘上的文件，能执行 shell 命令，能把结果写回文件。整个工作流在一个地方完成，不需要人来回搬运信息。

---

## 三家做的其实是同一件事

OpenAI 的 Codex CLI 用 Rust 写的，跑在本地终端，能读写整个代码仓库，还能运行测试、检查类型。Anthropic 的 Claude Code 类似，能在你的项目目录里直接编辑文件。Google 的 Gemini CLI 也是同样的思路。

三家的技术细节不同，但架构本质相同——都是把大模型从"对话框"搬到了"工作台"。

更有意思的是，三家不约而同都支持了 MCP 协议。MCP 是 Anthropic 提出的标准，让 AI 通过统一接口调用外部工具。

OpenAI 的 Codex CLI 支持把自己作为 MCP 服务器运行，也支持接入第三方 MCP 工具。这意味着命令行工具不只是一个"会写代码的聊天机器人"，而是一个可以接入任意工具的智能体运行时。

---

## 本质是从"回答"到"执行"

这件事放在更大的背景下看就更清楚了。

2023 年，ChatGPT 证明大模型能回答问题。2024 年，各家开始加文件上传、代码执行、联网搜索。2025 年，命令行工具出现——大模型第一次真正获得了在用户电脑上"动手"的能力。2026 年，Cowork 和 Codex App 进一步把这个能力从开发者推向了所有知识工作者。

每一步都在缩短"AI 给建议"和"事情做完了"之间的距离。

命令行是这个过程中关键的一步，因为它第一次把 AI 从浏览器的沙箱里放了出来。在浏览器里，AI 能看到的只是你贴给它的文字。在终端里，AI 能看到你的整个项目。

---

## 对我来说意味着什么

回到我自己的项目。我设计的那个水利多智能体平台，核心思路就是"固定工作流加灵活智能层"。仿真、控制、辨识这些核心模块是精雕细琢后固定下来的。AI 智能体在上层负责理解用户意图、选择工具组合、生成分析报告。

这个架构的设计过程本身，就是在命令行里用 AI 完成的。AI 帮我搜索最新的框架对比，帮我检查提示词的结构一致性，帮我自动更新章节编号。我只需要说"加入四预体系"，它就能把水利部的预报预警预演预案映射到具体的 Skill 和 Tool 链上。

这在网页版里做不到吗？理论上可以，但效率差距至少是五倍。

---

## 不只是开发者的事

最开始命令行工具确实是给开发者用的。但 OpenAI 最近把 Codex 从命令行扩展成了桌面应用，还加入了 Skill 的概念。你可以把一组指令和脚本打包成一个"技能"，让 AI 反复使用。Anthropic 的 Cowork 走得更远，直接在桌面上跑虚拟机，让非技术用户也能用。

这个演进方向很明确：先用命令行验证"AI 在本地执行任务"的模式，再把这个模式包装成普通人能用的界面。命令行是底层能力的试验场，桌面应用是面向大众的交付形式。

---

如果你还在只用网页版和 AI 聊天，建议试一试命令行工具。不是因为它更酷，而是因为当你让 AI 真正进入你的工作环境时，它能做的事情会超出你的想象。

从"帮我写一段话"到"帮我做完这件事"，中间差的就是一个终端。
