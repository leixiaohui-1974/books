## 一、核心设计思想：快思考与慢思考

HydroClaw 的架构灵感来自 Kahneman 的**双系统理论**：

| | 快思考（System 1） | 慢思考（System 2） |
|---|---|---|
| **HydroClaw 对应** | Skill（技能层 L3） | Agent（智能体层 L4） |
| **特征** | 固定路径、不可变、确定性、可验证 | 自主推理、灵活组合、创造性、需监督 |
| **水利类比** | 调度规程、标准操作流程 | 专家会商、应急研判 |
| **安全性** | 天然安全（流程固化） | 需 SafetyAgent 否决权约束 |
| **响应速度** | 毫秒级 | 秒~分钟级 |

**核心原理**：日常运行靠快思考（Skill），异常和创新靠慢思考（Agent）。两者的边界由 ODD（运行设计域）划定。

## 三、ch06 重构方案（核心章节）

### 新标题
**第六章 认知AI引擎：HydroClaw 的快慢思考架构**

### 目标字数：3.5万字

### 新学习目标

完成本章后，你应能够：

1. 用"快慢思考"类比解释 Skill（固定工作流）与 Agent（自主推理）的分工原理；
2. 阐述 MCP（Model Context Protocol）作为工具协议的设计哲学与优势；
3. 描述 HydroClaw "L0→L4" 五层架构的层间职责与数据流；
4. 设计四预闭环（预报→预警→预演→预案）的条件链接逻辑；
5. 解释 SafetyAgent ODD 三区间否决权的安全保障原理；
6. 理解认知API的"感知→认知→决策→控制"四维分类与CHS理论的对应。

### 新节次结构

```
6.1 从"会算"到"会想"：认知AI的角色                      [保留，重写]
    - T2a 解决了"算得准"（物理AI），T2b 要解决"想得清"（认知AI）
    - 认知AI三类核心能力：理解语义、解释决策、编排协同
    - 快慢思考隐喻：Kahneman System 1/2 → Skill/Agent

6.2 五层架构：从工具到智能体                             [★ 新增核心节]
    6.2.1 设计哲学：Tool → Skill → Agent 三级抽象
        - 工具（Tool）是原子的：做一件事，做好一件事
        - 技能（Skill）是固定的：步骤不可变，AI不可篡改执行顺序
        - 智能体（Agent）是自主的：可推理、可组合、需监督
    6.2.2 五层全景
        L0 Core       — 领域算法（14模块：仿真/控制/预测/ODD/水平衡/...）
        L1 Compute    — 分布式运行时（Ray）
        L2 MCP Servers — 工具服务（13服务器，~35工具）
        L3 Skills      — 固定工作流（17技能）
        L4 Agents      — 自主智能体（15个）
    6.2.3 层间原则
        - 上层只通过下层接口调用，不可跨层
        - 每层有独立的测试与验证体系
        - 故障隔离：某层失败不应导致相邻层崩溃
    图6-1: HydroClaw 五层架构原理图

6.3 MCP：让AI"用工具"的统一协议                         [★ 新增核心节]
    6.3.1 为什么需要工具协议？
        - LLM 本身不能直接计算水力学——它只能"调用"会算的工具
        - MCP 解决的问题：标准化工具描述/发现/调用/权限/审计
    6.3.2 MCP 协议的三要素
        - Tool Description: 工具做什么、输入输出是什么（JSON Schema）
        - Tool Discovery: 注册中心——AI 知道有哪些工具可用
        - Tool Invocation: 标准调用接口——输入→处理→输出→日志
    6.3.3 HydroClaw 的 MCP 实现
        - 13个 FastMCP 服务器，按领域分组
        - 工具分类示例：仿真类 / 预测类 / 安全类 / 评估类
        - 权限分级：哪些工具允许 AI 自主调用，哪些需人工确认
    6.3.4 工具调用闭环原理
        - 提问 → 意图分解 → 工具选择 → 执行 → 结果校验 → 回答
    图6-2: MCP 工具协议三要素
    表6-1: MCP 工具分类与安全分级原则

6.4 Skill：快思考——固定验证工作流                        [★ 新增核心节]
    6.4.1 Skill 的本质：把专家经验"固化"为不可变流程
        - Skill = 有序步骤序列 + 每步调用指定工具 + 条件分支
        - 不可变性：AI 不能添加/删除/调换步骤（与 Agent 的核心区别）
        - 可验证性：固定流程意味着可以做完整的 xIL 验证
    6.4.2 Skill 定义语言（YAML 元数据）
        - name, trigger_phrases, tools_required, max_execution_time
        - input_schema, output_schema
    6.4.3 四预闭环——CHS 核心快思考技能
        预报（Forecast）: 数据→模型→预测结果
        预警（Warning）:  预测 vs ODD边界 → 预警等级(none/yellow/orange/red)
        预演（Rehearsal）: 多方案仿真对比 → 方案排序
        预案（Plan）:     最优方案 → 应急调度指令
        四预闭环:         条件链接（warning ≥ yellow → 触发预演 → 触发预案）
    6.4.4 工业水网技能举例
        - 漏损诊断: 水平衡→异常检测→定位→验证
        - 蒸发优化: 冷却塔→焙烧→赤泥→总损→优化建议
        - 全局调度: 需求预测→蒸发预测→调度优化→ODD校验
    6.4.5 Skill 与 Agent 的分工红线
        - 规则：凡是能写成固定流程的，必须用 Skill 而非 Agent
        - 原因：确定性 → 可验证 → 可审计 → 可信任
    图6-3: 四预闭环条件链路图
    表6-2: 快思考（Skill）vs 慢思考（Agent）对比

6.5 Agent：慢思考——自主推理智能体                        [★ 新增核心节]
    6.5.1 什么场景需要慢思考？
        - Skill 无法处理的情形：未预见的组合异常、需要跨域推理、需要解释"为什么"
        - Agent 的能力：任务分解、灵活组合工具和技能、生成报告、与人对话
    6.5.2 HydroClaw 的七类域智能体（原理层面）
        - Orchestrator（编排）: 接收请求→意图分类→路由决策（4级路由）
        - Planning（规划）: 复杂任务分解为 DAG 依赖图
        - Analysis（分析）: 灵活的数据分析与诊断（"柔性"核心）
        - Report（报告）: 结果综合与可视化
        - Safety（安全）: ODD 边界守护 + 否决权
        - Handuo/瀚铎（认知）: RAG检索 + 知识图谱 + 领域问答
        - RL Dispatch（调度）: 强化学习策略 + 规则回退
    6.5.3 Orchestrator 的 4 级路由原理
        层级1: 技能触发短语匹配（最长匹配优先）→ 快思考
        层级2: 工具关键词匹配 → 直接工具调用
        层级3: 能力路由（Registry + 健康感知）→ 最佳Agent
        层级4: Planning分解（DAG）→ 多Agent协同
    6.5.4 SafetyAgent：ODD 三区间否决权
        - normal（绿区）→ 放行
        - extended（黄区）→ 需人工确认
        - mrc（红区）→ 否决 + 生成最小风险状态方案
        - 原理：安全Agent拥有对所有Agent的绝对否决权
    图6-4: Orchestrator 4级路由原理
    图6-5: SafetyAgent ODD三区间否决流程

6.6 RAG 与知识图谱                                      [保留，精简]
    6.6.1 RAG 原理：检索增强生成（Retrieval-Augmented Generation）
        - 核心价值：回答可追溯——LLM不能"凭空编"
    6.6.2 水利知识图谱
        - 工艺本体：实体（泵站/闸门/渠段）+ 关系（上游影响/联锁依赖）
        - 故障模式库：原因→效果映射
    6.6.3 HanduoAgent 的诊断闭环
        - 检索规程 → 匹配故障模式 → 模板回答 → LLM增强解释

6.7 认知API：感知—认知—决策—控制                         [★ 新增]
    6.7.1 CHS 理论的四维认知框架
        - 感知（Perception）: 数据采集与预测——"看到什么"
        - 认知（Cognition）: 预警与诊断——"意味着什么"
        - 决策（Decision）: 优化与设计——"怎么做最好"
        - 控制（Control）: 调度与执行——"具体怎么做"
```
